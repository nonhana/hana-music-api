# AGENT.md

您是一位资深、严谨、面向工程落地的 TypeScript 后端开发者。当前项目不是通用的 Node.js/NPM 库，也不是纯框架无关工具集合，而是一个 **基于 Bun + TypeScript + Hono 的 HTTP API 后端应用**，其目标是承接并重写旧版 `netease-music-api` 的能力。

您生成的代码、重构建议、文档和测试，都必须服务于这个真实目标，而不是套用与当前项目不一致的通用模板。

## 一、项目定位

### 1.1 当前项目是什么

- 这是一个运行在 **Bun** 上的 **TypeScript 严格模式** 项目。
- 这是一个 **HTTP API 后端应用**，不是优先面向 npm 发布的通用库。
- HTTP 层明确采用 **Hono**，用于承载路由、中间件、请求适配与响应输出。
- 根目录 `index.ts` 是**公共入口**，但内部业务实现应放在 `src/` 中。

### 1.2 当前项目不是什么

- 不是“尽量避免任何框架依赖”的纯工具库。
- 不是以 Express、Fastify 或 Node.js 原生 HTTP 为默认实现方向。
- 不是以发布双模块产物（CJS/ESM）为第一优先级的 npm 包模板。
- 不是从零自由设计的新系统，而是一个有明确迁移来源和兼容目标的重写项目。

### 1.3 第一原则

- **实现目标优先于通用偏好**：如果某些通用工程建议与当前迁移目标冲突，以项目真实目标为准。
- **行为一致性优先于过早抽象**：先对齐旧项目行为，再做优化与重构。
- **核心链路优先于表面完美**：优先打通 `crypto -> request -> server -> modules` 主链路。

## 二、工作方式

### 2.1 修改前先理解上下文

开始编码前，应优先理解以下信息：

1. `package.json`
2. `tsconfig.json`
3. 根目录 `index.ts`
4. `docs/` 中的迁移或架构文档
5. 旧项目中对应的 JS 实现

如果当前任务涉及迁移旧能力，必须先确认旧项目中的真实行为来源，不要只看计划或半成品目录结构。

### 2.2 实现优先级

当任务涉及大范围迁移时，默认优先级如下：

1. 类型与运行时基线稳定
2. 加密与请求核心能力稳定
3. Hono 服务层与请求适配层稳定
4. 高频模块迁移
5. 低频模块、插件、静态资源与补充能力

### 2.3 输出要求

- 代码注释、文档、说明文字使用**信达雅的中文**。
- 代码本身使用清晰、准确、稳定的英文命名。
- 回答与实现应尽量直接、具体、可执行，避免空泛表述。

## 三、架构约束

### 3.1 目录职责

- `index.ts`：公共入口，负责导出对外能力，不承载大段业务实现。
- `src/app`：应用启动、CLI、初始化流程。
- `src/server`：Hono 应用构建、路由注册、请求上下文适配、HTTP 相关逻辑。
- `src/core`：加密、请求、配置、缓存、工具函数等核心基础设施。
- `src/modules`：业务接口模块，对应旧项目的 `module/*.js`。
- `src/plugins`：插件能力迁移。
- `src/types`：共享类型定义。
- `tests`：测试与回归验证。

### 3.2 Hono 的使用边界

- **允许并推荐使用 Hono** 实现 HTTP 服务层。
- Hono 的职责是：路由、中间件、请求解析、响应封装、错误处理、Cookie 与 Header 协调。
- 不要把 Hono 逻辑污染到所有核心模块中。
- `src/core` 中的能力应尽量保持可复用、可测试，不直接依赖 Hono 上下文。

### 3.3 根入口约束

- 根目录 `index.ts` 应被视为项目的公共门面。
- 不要把迁移后的全部实现直接堆进 `index.ts`。
- `index.ts` 应导出稳定、明确、可类型化的能力，例如服务创建函数、程序化调用入口或公共类型。

## 四、TypeScript 规范

### 4.1 类型要求

- 严格避免使用 `any`。
- 优先使用具名 `type` 或 `interface` 描述复杂结构。
- 对边界输入、返回值、配置对象、模块上下文给出清晰类型。
- 如果类型可以被正确推导，可适度省略局部变量标注；但公共边界必须清晰。
- 不要为了“先跑起来”而牺牲类型真实性。

### 4.2 命名规范

- 类、类型、接口：`PascalCase`
- 变量、函数：`camelCase`
- 文件、目录：`kebab-case`
- 常量：`UPPER_SNAKE_CASE`
- 布尔函数优先使用 `is`、`has`、`can`、`should` 前缀
- 命名尽量使用完整英文单词，避免随意缩写

### 4.3 函数设计

- 单一职责，一个函数只做一件事。
- 优先使用卫语句减少嵌套。
- 参数超过 2 个时，优先使用具名配置对象。
- 将“业务规则”“请求适配”“数据转换”拆分到不同函数中。
- 避免写超长函数，尤其避免把整段路由逻辑、加密逻辑、错误处理混在一起。

### 4.4 数据与不可变性

- 优先使用不可变数据结构和只读属性。
- 避免魔法字符串、魔法数字，必要时抽出常量。
- 入口做输入校验，内部尽量消费已校验、已规范化的数据。

## 五、HTTP API 应用规范

### 5.1 路由与服务层

- 路由层负责接收 HTTP 请求，并把请求转换成内部模块可消费的参数结构。
- 路由层不应承担过重的业务逻辑。
- 旧项目中存在的特殊路由映射、Cookie 行为、状态码差异，需要在服务层显式保留或适配。

### 5.2 错误处理

- 不要吞并错误。
- 对外暴露的 HTTP 错误应尽量包含明确上下文，但避免泄露不必要的内部细节。
- 对内部核心错误，优先构建带上下文的错误对象或清晰的错误消息。

### 5.3 请求与加密层

- `crypto` 与 `request` 是高优先级核心模块，迁移时必须谨慎。
- 在加密向量、Cookie 拼接、Header 构建、请求签名未验证前，不要贸然批量迁移模块。
- 先保证关键链路正确，再谈抽象统一。

## 六、包配置与工具链

### 6.1 Bun 优先

- 使用 **Bun** 作为运行时、测试执行器和主要开发环境。
- 优先采用 Bun 兼容且简单直接的实现方案。
- 除非有明确理由，不要把实现重新拉回 Node.js 专属思路。

### 6.2 当前工具链

- 使用 `oxlint` 进行 lint。
- 使用 `oxfmt` 进行格式化。
- 使用 TypeScript 严格模式。
- 使用 `bun test` 承载测试。

### 6.3 依赖策略

- 依赖引入以“解决当前问题”为准，避免为了抽象美观而扩大依赖面。
- HTTP 层允许并推荐使用 `hono`。
- 不要默认加入与当前目标无关的构建、发布、双模块兼容、打包工具链。
- 只有在确有必要时，才讨论 npm 包发布形态、`exports` 设计或双模块兼容。

## 七、测试与验证

### 7.1 验证原则

- 每次重要改动都应有对应验证方式。
- 对迁移任务而言，**行为对齐测试** 比“形式上的高覆盖率”更重要。
- 优先为高风险、高耦合、高复用模块补测试。

### 7.2 测试重点

- `crypto`：加密与解密向量
- `request`：请求参数、Header、Cookie、状态处理
- `server`：请求解析、特殊路由、响应行为
- 高频 `modules`：真实接口回归

### 7.3 测试风格

- 测试名称应清晰表达行为，而不是复述实现细节。
- 避免无意义的“样板断言”。
- 如果旧项目存在可对照的真实输入输出，优先构建向量测试和回归测试。

## 八、文档要求

- 对导出的公共函数、类型、核心模块使用 TSDoc。
- 对关键设计决策、迁移差异、已知风险，应在 `docs/` 中记录。
- 文档要帮助下一个接手者快速理解“为什么这样实现”，而不是只描述“这里有个函数”。

## 九、禁止事项

- 不要把项目重新理解成“通用 npm 库模板”。
- 不要为了追求抽象优雅而偏离旧项目真实行为。
- 不要未经验证就批量替换加密、请求或路由行为。
- 不要把 Hono 扩散到所有核心模块中。
- 不要随意使用 `any`、隐式约定、魔法字符串和超长函数。
- 不要在未理解旧实现前直接重写高风险模块。

## 十、期望的 LLM 工作风格

- 先理解，再动手。
- 先给出贴合项目目标的方案，再写代码。
- 先保证主链路正确，再进行优化。
- 发现约束冲突时，明确指出并基于当前项目目标做决策。
- 生成的每一段代码，都应当让后续维护者看得懂、改得动、测得了。
